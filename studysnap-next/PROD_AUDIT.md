### [H1] Missing Rate Limiting on Authentication Endpoints
- **category/effort:** authz / small
- **where:** `app/api/auth` :: login/route.ts, signup/route.ts, request-reset/route.ts
- **impact:** Attackers can conduct unlimited brute-force attacks on user accounts, credential stuffing from breached databases, or account enumeration via password-reset endpoints without being throttled.
- **fix:** Implement IP-based rate limiting on all four auth endpoints. Options: (1) Use @vercel/kv with Ratelimit class (simplest for Vercel Hobby) — requires KV store setup. (2) Implement a lightweight in-endpoint rate limiter that tracks IP + endpoint in a database table (login_attempts or similar) with periodic cleanup. Suggested limits: login max 5/IP/15min, signup max 3/IP/hour, request-reset max 5/email/day (track by email not IP to prevent enumeration). Extract client IP via x-forwarded-for header (standard on Vercel). Add rate-limit response headers (Retry-After, X-RateLimit-*) per RFC 6585. Consider adding CAPTCHA to auth forms as secondary defense. Test with load testing to verify 429 responses under attack.

### [H2] Account Enumeration via Login Endpoint Timing
- **category/effort:** authz / small
- **where:** `app/api/auth/login/route.ts` :: lines 13-16
- **impact:** Account enumeration attack: attackers can programmatically determine which email addresses have valid accounts on the platform, reducing the search space for targeted attacks and enabling social engineering.
- **fix:** Apply constant-time mitigation to /api/auth/login/route.ts: either (1) fetch a dummy passwordHash from config/constants if user not found, then call bcrypt.compare() against both non-existent and existent user branches, OR (2) reorder logic to call bcrypt.compare() before returning early (using a computed dummy hash if user is null). Keep error message identical ('Invalid email or password') in both cases. Pattern is already correctly implemented in request-reset/route.ts (lines 17-20) — apply the same principle. Effort: ~10 minutes (3-4 lines of code change + config constant). Also: add rate limiting to /api/auth/login (e.g., 5 attempts per IP per 15 minutes) to further protect against enumeration.

### [H3] Password Reset Token Transport via URL Query Parameter
- **category/effort:** security / medium
- **where:** `app/api/auth/request-reset/route.ts` :: line 27
- **impact:** If a reset token URL is exposed (via browser history, logs, proxies, or email forwarding), an attacker can reset the user's password without knowledge of the current password. Reset tokens with 15-minute expiry help but don't fully mitigate.
- **fix:** Implement POST-based reset pattern:

1. Send a SHORT, disposable session ID (UUID) via email link, not the token itself. Example: `https://studysnap.com/reset-password?sid=<uuid>`

2. The reset-password page, when loaded with a valid `sid`:
   - Verifies the session ID server-side (store reset sessions in DB or Redis with 15-min expiry)
   - Renders the form without showing token in URL
   - On form submit, sends `{ sid, newPassword }` via POST to /auth/reset-password
   - Backend validates sid, then resets password

3. Alternative (OWASP-compliant): Send a short 6-digit code via email, user manually enters it on the reset page, then POST `{ code, newPassword }` - eliminates any magic link exposure entirely.

Why this works: The session ID has no cryptographic value on its own (can't be used offline to reset passwords). Only the combination of valid session ID + verified user device (via authenticated POST) allows password change. Tokens never appear in browser history or logs in cleartext.

Effort: Medium - requires:
- New reset_sessions table (or Redis prefix) with sid, user_id, expires_at
- Modify request-reset to create session instead of token
- Modify reset-password page to verify sid and use POST-only flow
- Update reset-password route handler to accept sid instead of token

Risk of change: Low - pure refactoring, no breaking changes to client contracts.

### [H4] Unchecked PPTX/DOCX/XLSX ZIP Extraction Without Decompression Bomb Protection
- **category/effort:** reliability / small
- **where:** `lib/pdf.ts` :: lines 60-84 (PPTX), lines 89-99 (XLSX)
- **impact:** An attacker can upload a specially crafted 15MB PPTX or XLSX zipbomb that decompresses to gigabytes of data, exhausting Node.js heap memory and causing an out-of-memory crash (DoS). This crashes the Inngest background function processing the file, affecting all users sharing that Vercel instance.
- **fix:** Implement extraction-time decompression bomb protection in lib/pdf.ts: (1) For PPTX/XLSX, track cumulative uncompressed size before calling .async('string') or sheet_to_csv(). (2) Set MAX_UNCOMPRESSED threshold (recommended 100-200MB, or lower if memory-constrained). (3) Abort extraction and throw a user-facing error if threshold is exceeded. Code example for PPTX: before the slide extraction loop, add `let totalUncompressed = 0; const MAX_UNCOMPRESSED = 100 * 1024 * 1024;` then in the loop check `totalUncompressed += zip.files[name].uncompressed; if (totalUncompressed > MAX_UNCOMPRESSED) throw new Error('File contents too large after decompression');`. For XLSX, track cumulative CSV string length during sheet iteration. Alternatively, run extraction in a worker thread with --max-old-space-size limit, but per-extraction checks are more efficient.

### [H8] Missing validation for pageSize parameter in history route
- **category/effort:** observability / small
- **where:** `app/api/history/route.ts` :: lines 10-11
- **impact:** Malformed query parameters cause a 500 error instead of a 400 validation error. If pageSize='abc', Prisma throws an error that becomes INTERNAL_ERROR.
- **fix:** Add explicit validation for integer constraints before passing to Prisma. Two approaches:

Option A (minimal): Add validation checks:
```typescript
const pageNum = Number(url.searchParams.get('page') ?? 1);
const pageSizeNum = Number(url.searchParams.get('pageSize') ?? 20);

if (!Number.isInteger(pageNum) || pageNum < 1) {
  throw new HttpError(400, 'INVALID_PAGE', 'page must be a positive integer');
}
if (!Number.isInteger(pageSizeNum) || pageSizeNum < 1 || pageSizeNum > 50) {
  throw new HttpError(400, 'INVALID_PAGE_SIZE', 'pageSize must be an integer between 1 and 50');
}

const page = pageNum;
const pageSize = pageSizeNum;
```

Option B (recommended): Use Zod with coercion for consistency:
```typescript
import { z } from 'zod';

const querySchema = z.object({
  page: z.coerce.number().int().min(1).default(1),
  pageSize: z.coerce.number().int().min(1).max(50).default(20),
});

const query = querySchema.parse(Object.fromEntries(url.searchParams));
const { page, pageSize } = query;
```

Option B is preferred because it centralizes validation, reuses the existing error handler (ZodError → 400), and sets a pattern for other routes.
